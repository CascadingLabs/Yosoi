# W2 — multi-profile cascade + per-IP vs per-profile block attribution

**Kind:** `answer` · **Confidence:** 0.74 · **Verifier verdict:** `needs-work`
**Full workstream:** W2 — multi-profile cascade + per-IP vs per-profile block attribution

## Gap (why Yosoi lost this on the hotpath)
Yosoi's browser tier is single-identity and stateless: the whole pool shares ONE BrowserConfig with no profile/user-data-dir, no proxy rotation, and no block-attribution. The waterfall escalates simple→headless→headful as a fetch-MODE ladder (yosoi/core/fetcher/waterfall.py:450-480) but headful spins up a FRESH anonymous Chrome (voiddriver.py:445-448, _browser_config_kwargs at :120-137 sets only headless/stealth/no_sandbox/chrome_executable + optional UA/locale) — it never reuses a logged-in/trusted profile. That is precisely why Nimbal bypassed Yosoi for Google: _real.py:13-14 hardcodes ONE trusted profile via raw `--user-data-dir` in PROFILE_ARGS and passes it as BrowserConfig.extra_args (_real.py:73,105). Two structural blockers in VoidCrawl make a cascade impossible today: (1) PoolConfig holds a SINGLE shared `browser: BrowserConfig` (__init__.py:257) and the native pool ctor `_PoolContext._from_params` (_ext.pyi:398-411) takes only scalar headless/proxy/extra_args + one extra_args list — so N tabs in a pool are ONE identity; multiple `--user-data-dir` cannot be expressed per-tab. (2) The clean profile-lease API (profiles.py with_profile/acquire_profile, _ext.pyi:676-698) is a SESSION primitive (ProfileHandle.new_page), NOT pool-backed, and is what _real.py should have used but didn't. On the attribution side, BotDetectionError (waterfall.py:435,464) carries no signal of WHICH identity (profile/IP) was blocked, and there is no proxy knob threaded from JSFetcher at all — block events are unattributable, so rotation has nothing to rotate on.

## Proposal
Add a profile-cascade primitive at the FETCHER layer (keeps replay runtime LLM-free, CAS-87 intact). Three pieces:

1) A `BrowserIdentity` dataclass (profile_dir | session-leased profile name, proxy, locale, timezone, geo) and an ordered `IdentityCascade` of N identities. Yosoi pins N identities and rotates to the next on a block, persisting the winning identity per-domain alongside the existing FetchStrategy cache (waterfall.py:184-191 _record_success is the exact hook — extend FetchStrategy with `identity_id`).

2) Because VoidCrawl's pool is single-identity (one shared BrowserConfig, __init__.py:257), DON'T fight it: model each identity as its OWN _VoidCrawlFetcher instance (its own pool/process with its own --user-data-dir via a real `extra_args` passthrough), OR for leased logged-in profiles route through the existing `with_profile` session API (profiles.py:89). The cascade owns a dict[identity_id, fetcher] and lazy-starts each like _ensure_headful (waterfall.py:261-267).

3) Block attribution: wrap BotDetectionError to carry `identity_id` + `captcha_kind` (Page.detect_captcha, _ext.pyi:463-469) and record (identity, domain, blocked_at). Rotation policy uses tenacity Retrying over the identity list (AGENTS.md mandate — no for/while sleep), `retry_if_exception_type(BotDetectionError)`, wait_exponential, reraise=True; each attempt binds the next identity. Fail-fast invariant preserved: cascade exhausted → raise, never a heuristic fallback.

Surface the cascade in BrowserConfig upstream too: add `user_data_dir: str|None` to VoidCrawl BrowserConfig (__init__.py:203-209) + a per-browser `browser_configs: list[BrowserConfig]` on PoolConfig so a pool can host heterogeneous identities natively — that's the clean long-term fix that retires Nimbal's raw-flag hack.

## Code sketch — the tangible change
```python
+# yosoi/core/fetcher/identity.py  (new leaf module, no eager re-export)
+@dataclass(frozen=True)
+class BrowserIdentity:
+    id: str
+    profile_dir: str | None = None      # --user-data-dir  (trusted/logged-in)
+    profile_name: str | None = None     # VoidCrawl leased profile (with_profile)
+    proxy: str | None = None            # egress IP
+    locale: str | None = None
+    timezone_id: str | None = None
+    geo: tuple[float, float] | None = None
+    headful: bool = False

# voiddriver._browser_config_kwargs — thread identity through (currently drops proxy/profile)
     kwargs = {'headless': self._headless, 'stealth': True, 'no_sandbox': self.no_sandbox,
               'chrome_executable': self.browser_executable_path}
+    if self._identity.proxy:        kwargs['proxy'] = self._identity.proxy
+    if self._identity.profile_dir: kwargs.setdefault('extra_args', []).extend(
+        [f'--user-data-dir={self._identity.profile_dir}', '--profile-directory=Default'])

# voiddriver — attribute the block
-    raise BotDetectionError(url, 200, indicators)
+    raise BotDetectionError(url, 200, indicators, identity_id=self._identity.id,
+                            captcha_kind=await tab.detect_captcha())

# waterfall — replace terminal headful with a tenacity-driven cascade
+    async def _fetch_cascade(self, url, domain, start_time, **kw) -> FetchResult:
+        last_winner = self._strategy_cache.get(domain)
+        order = self._cascade.ordered(prefer=last_winner.identity_id if last_winner else None)
+        async for attempt in get_async_retryer(max_attempts=len(order), wait_min=1.0,
+                wait_max=8.0, exceptions=(BotDetectionError,), log_callback=log_retry):
+            with attempt:
+                ident = order[attempt.retry_state.attempt_number - 1]
+                fetcher = await self._ensure_identity(ident)   # lazy per-identity pool
+                res = await fetcher._do_fetch(url, start_time, f'cascade:{ident.id}', **kw)
+                await self._record_success(domain, 'headful', identity_id=ident.id)
+                return res
+        raise BotDetectionError(url, 200, ['cascade exhausted'])  # fail-fast, no heuristic

# VoidCrawl __init__.py BrowserConfig (upstream clean fix)
+    user_data_dir: str | None = None   # native profile dir; retires extra_args hack
# PoolConfig
+    browser_configs: list[BrowserConfig] = Field(default_factory=list)  # heterogeneous identities
```

## Use cases unblocked (Nimbal SERP/authority)
- Google SERP rank/competitor scrape (nimbal _real.py:64-91 _human_search): needs headful + trusted profile; cascade pins [trusted_profile_headful, fresh_headful, proxy_A_headful] and rotates on reCAPTCHA so Yosoi finally owns the Google path Nimbal hand-rolled
- Bing/Brave/DDG SERP (todo.md: 'worked on plain headless'): cascade first identity = plain headless, zero profile — cheap path stays cheap, Google escalation is per-domain via _record_success identity_id
- Authority tools SemRush/Ahrefs/Moz/SimilarWeb (_real.py:95-127): SemRush needs the logged-in profile (LIVE_PROFILE comment 'SEMrush etc.'); leased profile_name identity via with_profile gives exclusive cookie reuse without copying user-data-dir
- 30+ drifting senior-living provider sites: per-domain identity persistence means most sites pin headless-no-profile and only the bot-walled few pay for trusted-profile/proxy identities
- Google Maps listing extract (yosoi_bridge MapsListingExtract): geo/locale identity (set_geolocation/set_locale, _ext.pyi:561-574) pins per-market consistency so place results match the target city/state

## Risks
- Memory blowup: each identity = its own browser PROCESS (pool can't share one BrowserConfig across profiles). scale.py per_tab_mb 80-120 + ~750MB fixed base PER PROCESS means N trusted-profile headful identities cost N×(750MB + tabs×~100MB). 3 headful identities ≈ 2.5-3GB before tabs. Cap concurrency: headful identities should be browsers=1, tabs_per_browser small (4-6), and the cascade must serialize headful escalation, not parallelize it.
- Profile lease contention: with_profile holds an EXCLUSIVE .voidcrawl.lock (profiles.py:1-13); two Yosoi workers wanting the same logged-in profile serialize/block (ProfileBusy). A logged-in trusted profile is a singleton resource — cascade must treat profile_name identities as a semaphore of 1, not pooled.
- Block-attribution ambiguity: a block could be per-IP OR per-profile OR per-fingerprint; without the isolation experiment below, rotating profiles on a per-IP block wastes the scarce trusted profile (gets it burned on a bad IP).
- Google trusted profile is a burn risk: reusing ~/.config/chromium logged-in cookies for automated SERP scraping risks the real account; should be a dedicated throwaway Google identity, not the user's daily profile (Nimbal's _real.py:13 points at the live ~/.config/chromium — already risky).
- Upstream BrowserConfig.user_data_dir + PoolConfig.browser_configs change touches the Rust _from_params binding (_ext.pyi:398-411) — bigger than a Python-only change; the Yosoi-side per-identity-fetcher approach ships without VoidCrawl changes and is the move-fast path.

## Direction (next step to ship)
Ship the Yosoi-layer cascade FIRST (no VoidCrawl change): new yosoi/core/fetcher/identity.py + per-identity fetcher dict in JSFetcher, BotDetectionError carrying identity_id, tenacity rotation, FetchStrategy gains identity_id. Run the isolation experiment to decide the rotation key BEFORE wiring proxies. Then upstream BrowserConfig.user_data_dir to retire Nimbal's raw-flag hack. EXPERIMENT (isolate per-IP vs per-profile on Google): two crossed trials against the Google SERP path. Trial A (hold IP, vary profile): fixed egress (no proxy / one proxy), loop K distinct profiles incl. one trusted + several fresh, fire the same N queries each, record first-block query index + captcha_kind per profile. Trial B (hold profile, vary IP): fix ONE profile, rotate K proxies, same query load, record block onset per IP. Read: if blocks track the profile regardless of IP → per-profile (rotate profiles, proxies don't help) → invest in profile farm. If blocks track IP regardless of profile → per-IP (rotate proxies, one good profile suffices) → invest in proxy pool. If both → joint identity (must rotate the PAIR). Instrument via existing detect_captcha (_ext.pyi:463) + a per-(identity, query_index) block log; serialize headful runs to respect the ~750MB/process budget. Keep replay runtime untouched (CAS-87).

## Open questions
- Does the live ~/.config/chromium profile (_real.py:13) survive concurrent leased access, or must we clone it to a throwaway dir per identity? VoidCrawl's lock is exclusive — likely need cloned copies, which multiplies disk + first-launch cost.
- Is there a proxy pool available to Nimbal/Yosoi at all? No proxy is threaded anywhere today (BrowserConfig.proxy exists but unused) — the per-IP arm of the experiment needs real egress IPs to vary.
- Should identity selection be per-domain (current FetchStrategy granularity) or per-(domain, query-intent)? Google may block branded vs demand queries differently (serp_template_queries render_with_intent), which would argue for finer-grained identity keys.
- What's the acceptable steady-state concurrency for headful trusted-profile identities given the ~750MB/process base — is 2-3 identities the ceiling on the target hardware, and does that throttle the 30+ site replay throughput?
