# W1 — mid-scrape reCAPTCHA as a behavior-tree reaction

**Kind:** `fix` · **Confidence:** 0.78 · **Verifier verdict:** `needs-work`
**Full workstream:** W1 — mid-scrape reCAPTCHA as a behavior-tree reaction (ReplayPlan → tree, deterministic LLM-free execute, off-hotpath learning patch)

## Gap (why Yosoi lost this on the hotpath)
The replay engine has no reaction/guard concept AND the recovery primitives it would need are not bound on the hotpath tab. Two stacked gaps: (1) `ReplayPlan` is a flat node list (yosoi/models/replay.py:107-110) and `execute_plan` is a flat `for node in plan.nodes` loop whose only response to an unexpected page state is `_fail`→raise (yosoi/core/replay/runtime.py:69-84, 95-98) — a mid-flow reCAPTCHA is a fatal error, not a tree event, and there is no way to react-then-resume. (2) VoidCrawl's real captcha recovery surface (`capture_captcha`/`inject_captcha_token`/`solve_captcha`) is Rust + MCP-server-only (crates/core/src/captcha.rs:101-196, crates/mcp_server/src/tools/actions.rs:377-549). The hotpath tab Yosoi actually acquires is `PooledTab` (yosoi/core/fetcher/voiddriver.py:200 `self._pool.acquire()`), and `PooledTab` exposes NEITHER `detect_captcha` NOR any solver method (voidcrawl/_ext.pyi:91-370). Even the linear `_real.py` path only gets `detect_captcha` because it uses `Page` via `BrowserSession.new_page` (_ext.pyi:463), not a pooled tab. So today a reaction subtree has no captcha API to call on the hotpath.

## Proposal
Make ReplayPlan a behavior tree of typed nodes and split execution into two strictly separated planes. PLANE A (hotpath, deterministic, LLM-free, CAS-87 preserved): a tree-walker `execute_tree` over composite nodes (SEQUENCE / SELECTOR / REACTION-decorator) whose leaves are ONLY the existing `_EXECUTORS` act kinds plus a small fixed set of recovery leaves built from primitives already on `PooledTab` (eval_js, dispatch_mouse_event, dispatch_key_event, inject-token-via-eval_js). A REACTION node wraps a child subtree with a `trigger` guard (a ReplayCondition) and a `recovery` subtree; the walker, before/after each child tick, evaluates active triggers — on fire it ticks the recovery subtree, and on recovery-success RESUMES the wrapped child instead of raising. PLANE B (off-hotpath, may use LLM/discovery): when no learned recovery handles a fired trigger, the walker returns a structured `ReactionMiss(trigger, captcha_info, ax_outline)` instead of raising; the discovery/orchestrator layer (mcp_orchestrator) solves it once (capture→solver/MCP→inject, or records a humanized-click recipe), then PATCHES the persisted tree by inserting a learned REACTION decorator around the node that was interrupted — so the next hotpath run handles it inline with zero model calls. Reuse the A3Node persistence pattern (voiddriver.py:358-436) verbatim for tree-patch save/amend/stats.

Resolve the captcha-API gap WITHOUT Playwright and without bloating the hotpath: ship the recovery leaves as pure-JS over `eval_js`. Detection = run VoidCrawl's own `CAPTURE_JS` (captcha.rs:201, already pure JS) via `tab.eval_js` → gives kind+sitekey+response_field_selector+widget_rect on a PooledTab. Token injection = port the `inject_captcha_token` JS (captcha.rs:158-193) to a leaf executor that runs it through eval_js. Humanized click (the "a3node" recipe) = a leaf that replays a recorded sequence of `dispatch_mouse_event` mouseMoved/Pressed/Released with stored offsets+jitter (primitives at _ext.pyi:327-351), no new binding needed. The actual solve (2Captcha/CapSolver network call) stays in PLANE B / discovery, never on the deterministic walker. Follow-up VoidCrawl ask: bind `detect_captcha`/`capture_captcha`/`inject_captcha_token` onto `PooledTab` so we drop the JS-port shim — surface as a VoidCrawl wrapper gap, not a blocker.

tenacity for the resume loop (no for/while sleep). Keep the `_EXECUTORS` table the single source of leaf truth; add a parallel `_RECOVERY_LEAVES` table so a new ActKind still can't silently fall through (preserve the test_runtime_dispatch.py invariant). Keep lazy `__init__` + .pyi in lockstep for the new model exports.

## Code sketch — the tangible change
```python
​# yosoi/models/replay.py — tree node model (additive; ReplayPlan stays back-comp)
class NodeKind(str, Enum):
    LEAF = 'leaf'           # wraps one ReplayNode (assess/act/expect) — today's node
    SEQUENCE = 'sequence'   # tick children L→R; fail-fast on first child failure
    SELECTOR = 'selector'   # tick children L→R; succeed on first child success
    REACTION = 'reaction'   # decorator: guard a child subtree with trigger→recovery

class TreeNode(BaseModel):
    kind: NodeKind
    id: str
    # LEAF:
    leaf: ReplayNode | None = None
    # SEQUENCE/SELECTOR:
    children: list['TreeNode'] = Field(default_factory=list)
    # REACTION (decorator over exactly one child):
    child: 'TreeNode | None' = None
    trigger: ReplayCondition | None = None      # e.g. AssertKind.CAPTCHA (new) / TEXT '/sorry/'
    recovery: 'TreeNode | None' = None          # subtree of recovery leaves
    resume: bool = True                         # re-tick child after recovery succeeds

class ReplayPlan(BaseModel):
    nodes: list[ReplayNode] = Field(default_factory=list)  # KEEP: legacy flat
    tree: TreeNode | None = None                            # NEW: behavior tree
    version: int = 2
    # compile(): if tree is None, wrap nodes in a SEQUENCE of LEAFs — one code path.

# new act kinds (recovery PRIMITIVES — closed set the tree composes / discovery patches with)
class ActKind(str, Enum):
    ...                       # existing
    CAPTCHA_PROBE = 'captcha_probe'     # eval_js(CAPTURE_JS) -> CaptchaInfo dict
    INJECT_TOKEN  = 'inject_token'      # eval_js(INJECT_JS, kind, token)
    HUMAN_CLICK   = 'human_click'       # replay dispatch_mouse_event recipe (the a3node)
class AssertKind(str, Enum):
    ...
    CAPTCHA = 'captcha'                 # trigger guard: CAPTURE_JS returned non-null

# yosoi/core/replay/runtime.py — hotpath walker (deterministic, LLM-free)
class ReactionMiss(Exception):          # signal to PLANE B, NOT a hard fail
    def __init__(self, trigger, info): self.trigger, self.info = trigger, info

async def execute_tree(tab, node: TreeNode, ctx: ReplayResult) -> Status:
    # check active reaction guards first — a captcha mid-flow is a normal tick event
    for guard in ctx.active_reactions:
        if await _condition_holds(tab, guard.trigger):
            if guard.recovery is None:
                raise ReactionMiss(guard.trigger, await _capture(tab))   # -> PLANE B learns
            for attempt in get_async_retryer(max_attempts=3, ...):       # tenacity, no sleep loop
                with attempt:
                    if await execute_tree(tab, guard.recovery, ctx) is OK:
                        break
            if not guard.resume: return OK
            # fallthrough: resume the wrapped child
    if node.kind is LEAF:      return await _tick_leaf(tab, node.leaf, ctx)  # reuses _EXECUTORS
    if node.kind is SEQUENCE:  ... fail-fast on first FAIL (today's semantics)
    if node.kind is SELECTOR:  ... succeed on first OK
    if node.kind is REACTION:
        ctx.active_reactions.append(node);
        try: return await execute_tree(tab, node.child, ctx)
        finally: ctx.active_reactions.pop()

_RECOVERY_LEAVES = {                      # parallel to _EXECUTORS; same no-silent-fallthrough test
    ActKind.CAPTCHA_PROBE: _exec_captcha_probe,   # await tab.eval_js(CAPTURE_JS_PORT)
    ActKind.INJECT_TOKEN:  _exec_inject_token,    # await tab.eval_js(INJECT_JS_PORT, kind, token)
    ActKind.HUMAN_CLICK:   _exec_human_click,     # replay dispatch_mouse_event sequence
}
# PLANE B (mcp_orchestrator): on ReactionMiss -> solve once -> tree.patch(insert REACTION
# decorator around interrupted node) -> persist via A3Node-style storage (amend stats).
```

## Use cases unblocked (Nimbal SERP/authority)
- Google SERP hammer (the named target): real_serp loops 30 queries (_real.py:78-85); when /sorry/ reCAPTCHA pops mid-loop, today the whole run dies. With the tree, a REACTION decorator around the per-query SEQUENCE fires on AssertKind.CAPTCHA, ticks recovery (cached human-click recipe or capture→solve→inject_token), resumes the SAME query, and the loop continues — Yosoi finally owns the SERP path _real.py:64-91 bypasses.
- Authority probes (Ahrefs/Moz/SimilarWeb, _real.py:95-127 + scrape_authority TOOLS): per-tool navigate currently degrades to status='blocked' on detect_captcha (_real.py:114-122). A learned per-domain REACTION turns each tool page into discover-once/replay-forever with inline captcha recovery instead of a brittle blocked-marker.
- Ahrefs lazy-mount Turnstile (capture_captcha widget_rendered=false case, captcha.rs:84-85): tree trigger fires only when CAPTURE_JS confirms a rendered widget, avoiding false-positive recoveries on runtime-loaded-but-unmounted walls.
- SEMrush (notes.md:5 — recaptcha+login+rate-limit): the SELECTOR composite lets one subtree try the logged-in profile path and fall back to a teleport/profile-rotation subtree, all without LLM on replay.
- Dynamic forward plan-building (notes.md:3, user steer): when a site exposes ~60 AX nodes (PooledTab.get_full_ax_tree _ext.pyi:172), PLANE B grows a SEQUENCE of click_by_role LEAFs from those primitives and persists it — the tree extends forward from the fixed primitive set.

## Risks
- SCALING: multiple anti-bot tabs (notes.md:8) — active_reactions guards add an eval_js(CAPTURE_JS) probe per tick; at 5+ concurrent PooledTabs × 30 queries this is real CPU/CDP traffic. Mitigate: only evaluate guards whose cheap precondition holds (e.g. URL contains /sorry/ or title check) before the full CAPTURE_JS; debounce guard eval to once per navigation.
- Hotpath purity (CAS-87): the resume loop must never call a solver/LLM. Enforce structurally — _RECOVERY_LEAVES contains only eval_js/dispatch primitives; the solve+token-fetch lives in PLANE B and is injected as a pre-resolved token act, mirroring how ys.llm is banned from _EXECUTORS (runtime.py:206-216).
- PooledTab has no captcha binding (_ext.pyi:91-370): the JS-port shim (CAPTURE_JS/INJECT_JS via eval_js) is a maintenance fork of captcha.rs — it will drift from VoidCrawl's Rust source. Pin a test that diffs the ported JS against captcha.rs, and file the VoidCrawl ask to bind these on PooledTab.
- Humanized-click recipe brittleness: recorded dispatch_mouse_event offsets (a3node) are coordinate-bound; widget_rect from capture (captcha.rs:62-68) shifts across layouts. Store offsets relative to widget_rect, re-probe rect at replay, not absolute page coords.
- Tree-patch persistence poisoning: a recovery that 'succeeds' but actually got soft-blocked could persist a bad patch (fail-fast invariant). Gate patch persistence on a post-recovery expect assertion (#search present / no captcha) before save, exactly like A3Node gates on min_content_length (voiddriver.py:411).
- Infinite resume: a trigger that never clears loops the recovery. tenacity stop_after_attempt caps it; on exhaustion raise (genuine fail-fast) — do NOT degrade to garbage.

## Direction (next step to ship)
1) Land the additive model in yosoi/models/replay.py (NodeKind/TreeNode + ReplayPlan.tree, version=2, compile() that wraps legacy flat nodes in a SEQUENCE — zero breakage). 2) Add execute_tree + ReactionMiss to runtime.py, keep execute_plan delegating through compile(); add _RECOVERY_LEAVES with the three pure-JS/dispatch leaves (port CAPTURE_JS + inject JS from captcha.rs verbatim into a yosoi/core/replay/_captcha_js.py constant module). 3) Wire the PLANE-B patcher in mcp_orchestrator (the _navigate_plan seam at :55) to catch ReactionMiss, solve once, and persist a learned REACTION via an A3Node-style store. 4) Regression test: a fake PooledTab whose eval_js returns CAPTURE_JS non-null on the Nth tick (simulating Google /sorry/ at query N) — assert tree reacts, resumes, completes the remaining queries, and a tree patch is persisted; second run handles it inline with zero ReactionMiss. Run `uv run poe ci-check` and keep yosoi/__init__.pyi in lockstep with any new public exports.

## Adversarial verification — `needs-work`
**Objections**
- WRONG HOTPATH — the load-bearing claim is misattributed. The proposal asserts the replay hotpath is execute_plan over ReplayPlan.nodes (runtime.py:69). It is not. The live VoidDriver hotpath replays via DOMLoader.replay(tab, node.domloader_acts) over an A3Node (voiddriver.py:399, 358-418); ReplayPlan/execute_plan is unwired scaffolding — confirmed by the proposal's OWN cited evidence (runtime.py:138 'nothing calls execute_plan yet'; mcp_orchestrator.py:61-67 'parked scaffolding... intentionally unwired'; my grep found zero non-test callers). So 'Yosoi finally owns the SERP path' via a ReplayPlan tree is building a reaction onto a code path that does not run in production. The mid-scrape captcha (notes.md item 3, VERY high priority) actually has to land on the A3Node/DOMLoader replay lane that voiddriver.py:200-242 executes, OR execute_plan must first be wired into VoidDriver — a prerequisite the proposal treats as already done. notes.md does say 'the ReplayPlan should be able to cope with this', so the user's intent points at ReplayPlan, but the spike must explicitly own the execute_plan→VoidDriver wiring as in-scope work, not assume it.
- SOLVER FRAMING IS FACTUALLY WRONG — VoidCrawl has NO 2Captcha/CapSolver network solver anywhere. solve_captcha (actions.rs:541-699) is a humanized checkbox click (RECT_JS locate widget → dispatch_mouse_event Move/Press/Release with deterministic ±1.5px jitter at :617-621) followed by a token-poll loop (:683-698). There is no external-solver code path in the repo. The proposal's entire PLANE-A/PLANE-B split is partly built on a false premise: it treats 'humanized click' (PLANE A leaf) and 'capture→solver→inject_token' (PLANE B network call) as two distinct mechanisms, when VoidCrawl's actual recovery IS the humanized click and inject_token is only meaningful if YOU bring an external solver that does not exist yet. inject_captcha_token (captcha.rs:152-196) only WRITES a token you already obtained; nothing in VoidCrawl obtains one. So INJECT_TOKEN as a hotpath leaf is dead unless the spike also builds a 2Captcha client — which is unstated scope.
- DUPLICATION RISK — the proposal proposes porting CAPTURE_JS + inject JS into a yosoi/core/replay/_captcha_js.py constant and replaying dispatch_mouse_event recipes by hand, i.e. re-implementing solve_captcha's humanized-click + RECT_JS + jitter logic (actions.rs:564-662) and capture_captcha (captcha.rs:101-134) in Python. That is a second copy of three non-trivial JS blobs that already exist and are maintained in Rust. The proposal acknowledges drift risk (risk #3) but understates it: it's not one JS blob, it's the whole solve sequence including the rect/offset/jitter geometry. The cleaner direction is the follow-up VoidCrawl ask (bind capture_captcha/inject_captcha_token/solve_captcha onto PooledTab) FIRST, not 'ship the JS-port shim now' — the shim is the expensive, drift-prone path and the binding is mechanical Rust (the functions already take &Page; PooledTab wraps a Page).
- code_sketch async-retry bug — `for attempt in get_async_retryer(max_attempts=3, ...)` is wrong; get_async_retryer returns an AsyncRetrying (retry.py:118) which requires `async for attempt in ...`. Minor, but it's the exact tenacity-correctness the invariant cares about, and a reviewer copy-pasting the sketch would write a non-iterable bug.
- GUARD-EVAL COST is real and the mitigation is hand-wavy. active_reactions runs eval_js(CAPTURE_JS) before/after every child tick. CAPTURE_JS (captcha.rs:201-264+) does ~8 querySelector calls per invocation. At 30 SERP queries × N concurrent PooledTabs, with a REACTION wrapping the per-query SEQUENCE, that is dozens of full-DOM probes per page. The proposed 'cheap precondition (URL /sorry/ or title)' is sound for Google specifically, but the proposal also wants the trigger to be a GENERIC predicate (open_question 2) for rate-limit/login walls that have no such cheap URL tell — so the cheap-precondition mitigation does not generalize to the very generality the proposal argues for.
- PooledTab is shared/pooled — a humanized-click recovery that calls dispatch_mouse_event moves the real pointer on a tab drawn from a shared pool (voiddriver.py:200 self._pool.acquire()). The proposal never addresses whether concurrent PooledTabs share input focus / a single CDP target, or whether headless dispatch_mouse_event even lands on the captcha checkbox without bringToFront (cf. MEMORY: voidcrawl headless screencast needs bringToFront). _real.py uses headless=False (_real.py:73,105) precisely because the humanized SERP path needs a real window; a headless pooled-tab humanized click may not reproduce that, and the proposal assumes parity.

**Invariant breaches flagged**
- No HARD invariant is breached by the design as stated — and that is a genuine strength worth recording: (1) Fail-fast preserved — ReactionMiss routes to PLANE B / discovery and on retry-exhaustion raises (no garbage fallback), matching _fail semantics (runtime.py:95-98); the patch-poisoning gate (risk #5, modeled on A3Node's min_content_length gate at voiddriver.py:411 / a3node.py) is the right guard. (2) CAS-87 replay-LLM-free preserved STRUCTURALLY — _RECOVERY_LEAVES parallels _EXECUTORS (runtime.py:217-226) and contains only eval_js/dispatch primitives; the solve/LLM lives in PLANE B. This correctly mirrors how ys.llm is banned from _EXECUTORS (runtime.py:209-216). (3) tenacity respected (get_async_retryer, retry.py:87) — but see the `async for` bug above. (4) No Playwright — recovery is eval_js + dispatch_mouse_event over VoidCrawl primitives only.
- LATENT CAS-87 risk to watch, not a current breach: PLANE B inserts a learned REACTION whose recovery subtree is composed of the fixed _RECOVERY_LEAVES. If a future patcher ever lets discovery emit an EVAL leaf carrying a model-derived dynamic script that's re-fetched at replay time, that quietly reintroduces non-determinism. Enforce that patched recovery subtrees may ONLY reference the closed _RECOVERY_LEAVES set + pre-resolved literal payloads (no live token fetch on the walker) — the proposal says this in prose (risk #2) but the model (TreeNode.recovery: TreeNode) does not type-enforce it. Consider a validator that rejects EVAL_LLM/EVAL_PYTHON anywhere under a recovery subtree.
- tenacity sketch bug (restated as a correctness flag, not an invariant violation): `for attempt in get_async_retryer(...)` must be `async for` — retry.py:118 returns AsyncRetrying.
