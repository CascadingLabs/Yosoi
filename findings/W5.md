# W5 — contract→ODM export + description-disambiguated redundant contracts

**Kind:** `answer` · **Confidence:** 0.82 · **Verifier verdict:** `needs-work`
**Full workstream:** W5 — contract→ODM export + description-disambiguated redundant contracts

## Gap (why Yosoi lost this on the hotpath)
Two gaps, both rooted in the same place: Yosoi treats a Contract as a *field-set* and forgets the class identity (name + docstring) past extraction.

NOTE #2 (the sharp one): `contract_signature()` hashes ONLY `field_descriptions()` (yosoi/utils/signatures.py:61-72) — field name + field description + yosoi_type. It ignores the class docstring AND the class name. So two contracts that differ only by docstring (AdLink vs OrganicLink, both {url,title}) produce a byte-identical signature. I ran it live: both hash to `d896394a38221c78` (collide:True). Because the per-domain selector cache (yosoi/storage/persistence.py:131-135, `contract_sig`), the lesson key (yosoi/storage/lesson.py:21, LessonKey.contract_signature in yosoi/models/replay.py:139-151), and the JS-script cache all key off (domain, contract_signature), the two contracts SHARE one cache slot and the second discovery clobbers the first — the exact failure already documented in nimbal scripts/serp_contracts.py:3-9 ("second pass clobbers the first"), which is why Nimbal collapsed to ONE SerpResults contract with a `sponsored` string field. Meanwhile the docstring IS preserved and reachable: `model_json_schema()['description']` returns it verbatim. So the disambiguator the user wants already survives into the schema; the signature just throws it away.

NOTE #1: Yosoi's terminal output type is `ContentMap = dict[str, object]` (yosoi/core/pipeline/utils.py:28 and 5 sibling modules). It deliberately stops at the dict (yosoi_bridge.py:6-15), forcing every consumer to hand-write a `dict->model` adapter (the entire enrichment/yosoi_bridge.py layer-2). There is no contract->ODM/model export path in the repo (grep for beanie|mongo|django|ninja|model_json_schema in yosoi/ finds only MCPDiscoveryDraft.model_json_schema for LLM output formatting, nothing for export). A Contract is already a pydantic BaseModel carrying full field types + yosoi_type + description in its JSON schema, so the export is mechanically available — it's just never offered.

## Proposal
NOTE #2 — disambiguate redundant contracts by description, one blessed path:
Fold the class docstring into contract_signature. It's a 2-line change at the one chokepoint and immediately gives AdLink/OrganicLink distinct cache slots, distinct lessons, distinct cached selectors on the SAME domain — no second knob, consistent with description=-over-hint=. Then add a thin `Contract.variant(name, description)` classmethod (a create_model wrapper, mirroring to_selector_model/ContractBuilder) so you DECLARE near-identical contracts without copy-paste, and fix the __name__ registry clobber by keying the registry on (name, signature) or namespacing. Real use: on a google.com SERP, declare AdLink and OrganicLink with identical {url,title} fields but different docstrings; Yosoi discovers + caches a DIFFERENT selector for each (the ad-rail container vs the organic list) instead of one overwriting the other — recovering the multi-block capability serp_contracts.py had to abandon.

NOTE #1 — contract -> ODM export, no hand-written adapter:
Add `Contract.to_model(base=BaseModel, *, name=None, settings=None, include=None)` that uses pydantic.create_model to project the contract's fields onto an arbitrary base class — pass `base=beanie.Document` for Mongo, a Django-Ninja Schema, or plain BaseModel. yosoi_type/description ride along in json_schema_extra automatically (proven: model_json_schema carries them). This is native + serializable: no plugin, no import of beanie inside yosoi (base is injected by the caller, keeping lazy-loading clean and not adding a dep). It does NOT replace yosoi_bridge.py's *semantic* layer-2 work (status enums, Level granularity, run_id stamping — those are genuine consumer decisions, correctly outside Yosoi per its fail-fast/no-decide charter) but it DOES delete the boilerplate field-restating half: the consumer base class declares only the extra envelope fields and inherits the extraction fields. Respect fail-fast: export is a pure type projection, adds zero heuristics.

## Code sketch — the tangible change
```python
--- yosoi/utils/signatures.py  (NOTE #2 core fix — fold docstring in)
 def contract_signature(contract: type[Contract]) -> str:
     field_descs = contract.field_descriptions()
     field_sigs = sorted(field_signature(name, desc, _get_yosoi_type(contract, name))
                         for name, desc in field_descs.items())
-    payload = json.dumps(field_sigs)
+    # Class docstring is a load-bearing disambiguator: two contracts with identical
+    # fields but different NL intent (AdLink vs OrganicLink) MUST get distinct cache
+    # slots, not clobber each other (see nimbal serp_contracts.py lesson).
+    payload = json.dumps({'doc': _normalize(contract.__doc__), 'fields': field_sigs}, sort_keys=True)
     return hashlib.sha256(payload.encode()).hexdigest()[:16]

--- yosoi/models/contract.py  (NOTE #2 — declare redundant variants, one path)
 @classmethod
 def variant(cls, name: str, description: str) -> type[Contract]:
     """Redundant sibling differing ONLY by NL intent (the disambiguator).
        OrganicLink = Link.variant('OrganicLink', 'A free/organic search result link.')
        AdLink      = Link.variant('AdLink', 'A paid/sponsored result link.')
        Same fields -> now DISTINCT signatures -> distinct per-domain selectors."""
     sub = pydantic.create_model(name, __base__=cls)  # inherits fields
     sub.__doc__ = description
     return sub

--- yosoi/models/contract.py  (NOTE #1 — native ODM/model export, base injected)
 @classmethod
 def to_model(cls, base: type[BaseModel] = BaseModel, *,
              name: str | None = None, include: set[str] | None = None,
              **extra_fields: Any) -> type[BaseModel]:
     """Project this contract's fields onto `base` (beanie.Document / Ninja Schema / BaseModel).
        yosoi_type + description ride along via json_schema_extra. No adapter, no plugin."""
     defs = {n: (fi.annotation, fi) for n, fi in cls.model_fields.items()
             if include is None or n in include}
     defs.update(extra_fields)              # caller's envelope (run_id, captured_at, _id...)
     return pydantic.create_model(name or f'{cls.__name__}Model', __base__=base, **defs)

# consumer (replaces the field-restating half of yosoi_bridge / serp.py swap):
#   import beanie

#                run_id=(str, ...), captured_at=(datetime, ...))
#   class MapsDoc.Settings: name = 'maps_listings'   # set post-hoc or via base mixin
```

## Use cases unblocked (Nimbal SERP/authority)
- Google/Bing SERP: declare AdLink + OrganicLink (identical {url,title}, distinct docstrings) and have Yosoi cache a DISTINCT selector per block on one domain — recovering the multi-block extraction serp_contracts.py had to collapse into a single sponsored-string hack
- Local-pack vs organic vs ai_overview-source links on google.com — three description-only variants over {url,title} mapping straight onto serp.py's Block Literal
- MapsListingExtract -> to_model(beanie.Document): turns nimbal's aspirational 'swap to beanie.Document' comment (serp.py:7) into one call, deleting the field-restating boilerplate while keeping the genuine status/Level de-bias in yosoi_bridge layer-2
- SeoRowExtract / DataForSEO ranked-keyword rows exported directly to a Django-Ninja Schema for the API surface without a second hand-written model

## Risks
- Changing contract_signature INVALIDATES every existing cached lesson/selector (signature is the cache key) — it's a one-time cold re-discovery across all 30+ domains. Acceptable for a spike but must be flagged; consider a signature version bump so stale lessons mark STALE not silently miss
- Docstring-in-signature makes signatures sensitive to doc edits — a typo fix in a docstring now triggers re-discovery. _normalize() (lowercase+collapse-ws) softens this but doesn't eliminate it. Trade-off: that's the POINT for note #2, but it surprises authors who treat docstrings as free-form
- to_model with base=beanie.Document risks dragging beanie/pymongo into import graph IF anyone imports it eagerly — keep base caller-injected and NEVER add beanie to yosoi deps or lazy _LAZY map; document that the consumer owns the base import
- _CONTRACT_REGISTRY keyed by __name__ still clobbers if two variants get the same name; variant() must enforce unique names or the registry needs (name,sig) keying — otherwise resolve_contract by string name is ambiguous
- Beanie Document fields with the same name as pydantic-reserved/ODM internals (id, revision_id) could collide with contract fields — to_model needs a guard or include= to scope

## Open questions
- Should contract_signature also incorporate the class __name__, or only the docstring? Name-based would let two identically-documented-but-differently-named contracts split; docstring-only is the user's stated axis. Recommend docstring-only to honor 'differ ONLY by description'.
- Cache migration policy: hard-invalidate (delete) stale lessons, or mark STALE and lazily re-discover on next replay-miss? Latter is safer for 30 live domains but needs a signature-version field on LessonKey.
- Does to_model need to strip yosoi_action (ys.js/ys.File) fields from the exported ODM model, or carry them as plain typed fields? Action fields have no selector but DO have a real value at runtime — likely keep them.
- Where should Beanie's `class Settings: name=...` come from — a to_model(settings=...) kwarg, or left to the consumer to attach post-hoc? A kwarg is cleaner but couples the signature to beanie's API shape.
