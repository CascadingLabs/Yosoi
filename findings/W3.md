# W3 — teleport double-load + multi-engine SERP latency + geopy

**Kind:** `fix` · **Confidence:** 0.78 · **Verifier verdict:** `needs-work`
**Full workstream:** W3 — teleport double-load + multi-engine SERP latency + geopy

## Gap (why Yosoi lost this on the hotpath)
Yosoi's replay teleport (`runtime.py:360 _teleport`) fires `set_geolocation/set_timezone/set_locale` one-shot with NO ordering contract relative to the first navigate, and `_exec_teleport` is just one more node in `plan.nodes` — so whether the spoof lands before first paint depends entirely on where discovery emitted the TELEPORT node in the plan. The "double-load" the notes blame on teleport is NOT a VoidCrawl reload requirement: the Rust emulation tests prove `set_geolocation` then a single `navigate` then read works in one load (emulation.rs:80-107, 47-75). CDP `Emulation.setGeolocationOverride` is a session-level override that persists across navigations (page.rs:901-927) — it does not need a reload. The real double-load is two SEPARATE engine-side effects that live in Nimbal scripts, never modeled in Yosoi: (1) one-zone lag — Google/Bing/Brave resolve location on load N and only apply it on request N+1 (geolocation_pizza_search.py:16-20), and (2) the secure-context proof-read ritual (goto example.com just to read navigator.geolocation). Yosoi has no notion of either, so a discovered SERP plan that teleports mid-stream gets stale (IP-based) results on first read with no fail-fast signal.

## Proposal
Three changes, all LLM-free and fail-fast-preserving:

1) FIX THE ORDERING (teleport-before-first-paint). Make teleport a pre-navigation page-setup act, not a free-floating node. Two sub-options: (a) keep TELEPORT as an act but have discovery ALWAYS emit it as node[0] before the first NAVIGATE, and add a fail-fast model_validator that a plan containing TELEPORT places it before any NAVIGATE; (b) better: add an optional `ReplayPlan.teleport: TeleportSpec | None` applied by `execute_plan` BEFORE the node loop, so the CDP override is guaranteed live before first paint. This matches emulation.rs (override→navigate→read in ONE load) and kills the secure-context proof ritual entirely — we never need the example.com prime because we trust the CDP test, not navigator.geolocation read-back.

2) MODEL THE ENGINE-SIDE one-zone lag explicitly (this is the ACTUAL double-load) rather than letting it leak as a mystery. Add `ReplayAct.metadata['prime_reload']: bool` for NAVIGATE acts targeting known lag-y engines; the navigate executor does goto→goto when set. This keeps the double-load DETERMINISTIC and visible in the plan instead of a magic sleep. Pair with a fresh-tab-per-zone contract surfaced as a VoidCrawl session boundary (cookie stickiness, pizza example:11-14).

3) INTEGRATE geopy as a DISCOVERY-TIME helper, NOT a replay-time dep. geocode(city,state)->(lat,lon) runs when building the plan's TeleportSpec; the persisted plan carries literal lat/long, so replay never imports geopy. Lazy-load geopy behind a function (it pulls geographiclib + network); guard the import inside the helper. Mirror Nimbal's cache + RateLimiter(1.1s) + territory-strip. Cross-feeds Nimbal EstimateInput.latitude/longitude (model.py:29) and the serp.py 'teleport vs words' axis: teleport plans get a TeleportSpec, words plans get city/state baked into the query string with no TeleportSpec.

LATENCY MEASUREMENT PLAN for multi-engine SERP: instrument per (engine, phase) where phase ∈ {prime_goto, read_goto, dom_settle, extract}. Replace serp_band.load_twice's flat asyncio.sleep(1.0) with a tenacity Retrying that polls a 'location-applied' selector/text predicate (e.g. city name in body, or local-pack present) up to a deadline — turns the blind sleep into a measured wait and gives a per-engine settle-time distribution. Brave/Bing slowness is then a number, not a vibe. Budget target: drop the fixed 2.0s inter-query sleep in _real.py:85 once we have measured settle times.

## Use cases unblocked (Nimbal SERP/authority)

## Risks
- Engine cookie stickiness (pizza:11-14): a pooled/reused VoidCrawl tab leaks zone N's location into N+1; replay needs a fresh-session-per-zone contract or the multi-location band is silently wrong — scaling risk across 30+ sites sharing a tab pool
- prime_reload doubles navigations on the hottest path — at 30 sites x 3 engines x N intents that is a real latency/quota multiplier; must be gated to only the engines that actually lag (measure first, don't blanket-enable)
- Nominatim TOS: 1 req/s hard cap + requires a real User-Agent; bulk geocoding 30+ territories cold will rate-limit — the on-disk cache is load-bearing, and a shared cache across Yosoi+Nimbal risks divergence
- geopy/geographiclib added to Yosoi deps even if lazy: weight + a new transitive surface; if it ever gets imported at replay time it breaks the import-light invariant the lazy-loading rules protect
- Removing the example.com secure-context proof-read trusts the CDP override blindly; if a future VoidCrawl/Chromium change regresses setGeolocationOverride, replay would teleport-fail silently — needs a cheap one-time assert in discovery, not replay
- headless=False in _real.py:73,105 — if the resilient replay path must stay headless (unattended), teleport/local-pack behavior may differ from the headful capture scripts; verify parity before trusting captured plans

## Adversarial verification — `needs-work`
**Objections**
- GAP IS REAL but its severity is overstated as 'in production'. There is NO caller of execute_plan anywhere (runtime.py:138 NOTE, mcp_orchestrator.py:62-66 confirm it is intentionally unwired), and discovery only ever emits a single NAVIGATE node (mcp_orchestrator.py:70 — `ReplayAct(kind=ActKind.NAVIGATE, url=url)`). Grep finds ZERO `kind=ActKind.TELEPORT` emissions in the whole codebase. So teleport ordering is not 'broken in production' — teleport replay has never run at all. The proposal's framing ('a discovered SERP plan that teleports mid-stream gets stale results') describes a hazard for code that does not yet exist. That is fine as a forward-looking design, but it must be stated as 'design the contract before discovery emits TELEPORT', not 'fix a live bug'.
- The central technical premise — 'set_geolocation alone lands the SERP spoof in one load, the example.com prime is unnecessary' — is CONTRADICTED by VoidCrawl's own source. page.rs:896-898 explicitly documents: 'Header/IP-driven geo (e.g. Google Maps) keys off set_locale and the request URL more than this [geolocation override].' emulation.rs:80-107 only proves navigator.geolocation JS readback returns the override coords — it does NOT prove Google/Bing/Brave SERP localization (which is IP/header/cookie driven, not navigator.geolocation driven) changes from one set_geolocation call. The pizza example's own comments (lines 11-20) say engines resolve on load N and apply on N+1 AND write a sticky cookie — i.e. the engine does NOT read navigator.geolocation at all. So 'we trust the CDP test, not navigator.geolocation read-back' is a category error: the CDP test exercises a DIFFERENT mechanism than the one the SERP engines actually use. Removing the prime/proof on the strength of emulation.rs is unjustified.
- Proposal #2 (prime_reload = goto→goto inside _exec_navigate) only models the one-zone LAG. It silently drops the OTHER, co-equal cause the pizza example documents first (line 11-14): cookie STICKINESS — the engine writes the first resolved location into a cookie and reuses it for the whole session, so a reused tab leaks zone N into zone N+1. A double-goto on the SAME tab/session does nothing about a cookie that was already written on a prior zone's first goto. Without the fresh-session-per-zone boundary, prime_reload on a pooled tab can still serve stale results. The proposal mentions this as a 'risk' but its concrete code change does not address it — the fix is incomplete as drawn.
- The 'latency measurement' centerpiece — replace asyncio.sleep(1.0) with a tenacity Retrying that polls a 'location-applied' predicate — lives in nimbal/scripts/serp_band.py:102-114, which is NIMBAL hotpath code, not Yosoi. Rewriting Nimbal's load_twice is out of W3's stated Yosoi scope and conflates two repos. More importantly, the equivalent Yosoi mechanism ALREADY EXISTS: ReplayCondition with AssertKind.DOM_STABLE / TEXT / COUNT (replay.py:41-51, runtime.py:229-250) is exactly a deadline-bounded 'wait until predicate' on the expect step. The right Yosoi move is to express settle-as-condition on the node's `expect`, not to invent a parallel tenacity polling loop — the proposal reinvents the node assert machinery it should be using.
- Confidence 0.78 is too high given the load-bearing claim (one set_geolocation = correct SERP locale) is unverified against the actual engine mechanism and is actively warned against by page.rs:896-898. The proposal's own open_question #2 admits 'pizza example only verified DDG+Google' and the direction admits an experiment is still needed — so the confidence should be well below 0.78 until that experiment runs.

**Corrections to fold in (survived scrutiny)**
- Reframe #1 from 'fix a live ordering bug' to 'establish the teleport contract before discovery emits any TELEPORT node'. The ReplayPlan.teleport pre-loop field (sub-option b) is the correct shape and is genuinely better than node[0]-ordering, because: (a) it is structurally impossible to place after a NAVIGATE, (b) it needs no fail-fast model_validator scanning node order, (c) it matches the per-plan/per-zone reality (open_question #3 — one teleport per plan). Add a model_validator on ReplayPlan that, IF plan.teleport is set, the TeleportSpec carries non-None latitude AND longitude (fail-fast — this directly fills the replay.py:64-94 gap where no validator checks teleport coords today).
- Keep the example.com secure-context proof and the prime load as DISCOVERY-TIME verification, do NOT delete them from the design. The cheap correct move: discovery, when building a TeleportSpec, runs override→navigate→read-back (navigator.geolocation) ONCE to assert the override mechanism works, AND separately confirms the target engine's SERP actually localized (e.g. local-pack/city-name predicate) — recording in the lesson whether that engine needed prime_reload. Replay then trusts the recorded flags. This preserves fail-fast (a silent geolocation regression is caught at discovery, surfaced as a VerifyReport failure) without putting the proof on the replay hot path.
- Make prime_reload AND fresh-session boundary a single coherent unit, not two half-measures. Model zone isolation as a real plan/lesson property: one ReplayPlan per zone, executed against a freshly-opened voidcrawl session (session_open per zone, session_close after), so the cookie jar is clean — this is what the pizza example's fresh-BrowserSession-per-zone (lines 11-14, 118-120) actually does. prime_reload then only fixes the residual one-zone lag WITHIN a clean session. Without the session boundary, prime_reload is necessary-but-insufficient.
- Express settle-time as the node's `expect` ReplayCondition (AssertKind.TEXT for city-name-in-body, or AssertKind.COUNT for local-pack rows, with timeout_ms as the deadline) — reuse runtime.py:229-250, do not add a separate tenacity polling loop in Yosoi. The deterministic, fail-fast wait is already the node assert. If a measured per-engine settle distribution is wanted, instrument the existing _wait_for_dom_stable / condition path and emit timings; keep the tenacity-rewrite confined to Nimbal's serp_band.py as a separate Nimbal-scope task.
- Geopy-at-discovery-time with lazy import and literal coords baked into the persisted TeleportSpec (proposal #3) is the one part that is fully correct and invariant-safe — keep it as-is, mirroring nimbal/core/geocode.py:51-75 (cache + RateLimiter(1.1s) + territory-strip + None→words-plan routing). One addition: the geopy import guard belongs behind the discovery helper function body (as sketched), and geopy must NOT be added to any replay-path module import; add a test asserting `import yosoi.core.replay.runtime` does not transitively import geopy, to enforce the import-light invariant.
